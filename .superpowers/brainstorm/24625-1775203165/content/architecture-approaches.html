Architecture Approach

How should we structure the codebase and 3D engine integration?

A

Recommended: Zustand-First with R3F

All game state lives in Zustand stores. React Three Fiber renders the 3D scene reactively. No game loop — Zustand state changes trigger re-renders.

Pros

  • Simplest mental model — state in, visuals out
  • React devtools work for debugging
  • Easiest to understand for React developers
  • Card/deck/perk logic is pure JS functions

Cons

  • useFrame() still needed for movement/physics/VFX
  • Must be careful not to cause unnecessary re-renders
B

Game Loop + Zustand Hybrid

A central useFrame() game loop drives everything. Zustand stores hold state but the loop reads/writes it each frame. Closer to traditional game architecture.

Pros

  • More control over update order
  • Familiar to game developers
  • Easier to add timeScale (slow-mo death)

Cons

  • More complex — two update systems (React + loop)
  • Harder to debug state flow
  • More boilerplate code
C

ECS (Entity Component System)

Use an ECS library (like miniplex) for entities. Enemies, spells, player are all entities with components. Most scalable but most complex.

Pros

  • Scales to hundreds of entities efficiently
  • Clean separation of concerns
  • Industry-standard game pattern

Cons

  • Steep learning curve — new paradigm to learn
  • Overkill for MVP scope (~20 enemies max)
  • Extra dependency, more concepts to debug

My Recommendation: Option A

For your MVP with tactical pacing and ~20 enemies, Zustand-first is the sweet spot. It keeps things simple — card logic and game state are plain functions, and R3F handles rendering reactively. We use useFrame() only where we must (movement, collisions, VFX particles). This matches your preference for simplicity and is easiest to debug.
